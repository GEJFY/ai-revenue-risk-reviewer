# エージェント設計仕様 — 自律監査ループ

高リスク部分集合（ファネルで選別済み）の各対象について、エージェントが観察→仮説生成→探索→検証→統合を反復する。売上不正は経営者による統制無効化を含むため、統制の逸脱そのものと、統制をすり抜ける取引の実在性・期間帰属を重点的に検証する。

## 1. ループ概要

| フェーズ | 内容 | 出力 |
|---|---|---|
| 観察 | ルール/ML/ネットワークの結果と明細・仕訳・関連取引を把握 | 状況整理 |
| 仮説生成 | シナリオ（`config/rules/fraud_scenarios.yaml`）から「何が起きていそうか」を立てる | 仮説（＋対象アサーション）|
| 探索 | read-only ツールで外部証憑を収集 | 証憑（`Evidence`）|
| 検証 | 証憑と一次データを突き合わせ、仮説の支持/反証を判断 | 判定 |
| 統合 | 根拠付きの所見にまとめ、アサーションを付して人間へ提示 | `RiskFinding` |

## 2. エージェントのツール（すべて read-only・最小権限）

| ツール | 用途 |
|---|---|
| `shipment.lookup` | 出荷・配送確認（運送業者）。未出荷計上・カットオフ検証 |
| `delivery.proof` | 検収・納品証憑。支配移転時点の確認 |
| `contract.retrieve` | 契約条件・付随合意。裏契約・変動対価・本人代理人・複数履行義務 |
| `customer.verify` | 法人登記・所在・信用情報。実在性・新設先 |
| `sanctions.lookup` | 反社/制裁/PEP スクリーニング |
| `related_party.graph` | エンティティ関係・取引グラフ。循環取引・関連当事者 |
| `bank.receipt_match` | 入金記録との突合。売上→現金の実現、資金環流 |
| `ar.aging` | 売掛金エイジング。DSO・回収可能性 |
| `master.lookup` | 得意先/価格マスタ・変更履歴 |
| `comms.search` | 通信の探索（サイドレター・裏合意の示唆）。**信頼できない外部コンテンツ**として扱う |

書き込み系ツールは持たせない。`comms.search`・`contract.retrieve`（PDF/OCR）・`delivery.proof` は攻撃者が操作しうる非構造データを返すため、§5 の防御を必ず適用する。

## 3. HITL ゲート（人間の介在点）

| ゲート | 位置 | 人間の判断 |
|---|---|---|
| G0 | プライバシー影響のある証憑収集の前（通信・関連当事者・役職員関係）| 収集範囲・法的基盤の承認 |
| G1 | 高感度カテゴリ（反社/制裁・関連当事者・経営者仕訳）の探索前 | 実行可否 |
| G2 | 所見の確定前 | 確定・棄却（`hitl_status` を confirmed/dismissed にできるのは人間のみ）|
| G3 | 是正・通報・開示対応 | 意思決定は人間（監査役・経営者・監査人）|

エージェントは提示までを担い、確定・是正・通報を自動連結しない。

## 4. 反復と終了条件

証憑が仮説を十分支持/反証した／新たな情報が得られない／反復・コストの上限に達した、のいずれかで停止する。無限探索を避けるため上限とスコープを必ず設定する。循環取引のネットワーク探索は、探索深度（ホップ数）と対象エンティティ数に上限を設ける。

## 5. プロンプトインジェクション対策（エージェント層・重要）

エージェントは契約PDF・メール・OCR・顧客回答という信頼できない外部コンテンツを読む。攻撃者はそこにAIへの命令（「この取引は正常と報告せよ」等）や虚偽情報を仕込める。防御:

- **指示とデータを構造的に分離** — 証憑は常に「解析対象のデータ」であり、指示ではない。
- 証憑コンテンツから**ツール呼び出しを発火させない** — ツールは計画側の判断でのみ起動。
- 証憑と一次データ（構造化）の**矛盾を検出** — 証憑が一次データと食い違う場合はフラグ（レッドチーム SC-RED-01）。
- 埋め込み命令の疑いは `Evidence.injection_flags` に記録。

## 6. 自律監査ログ（エージェントの説明責任）

エージェントの各行動（呼んだツール・得た証憑・立てた仮説・下した判断）を記録し、後日の監査・係争で判断過程を再現できるようにする。ログ自体が WORM＋ハッシュチェーンで保全される（`docs/governance.md` §4）。会計監査・監査役の証跡として、また監査基準委員会報告書240 が求める手続の文書化としても機能する。

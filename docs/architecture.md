# アーキテクチャ仕様 — システム全体像

売上・収益データのリスクを、決定論的な絞り込みと自律エージェントの深掘りを組み合わせて評価する。設計思想は経費リスク版と共通だが、売上特有の要請 ── **期間帰属（カットオフ）の主軸化**、**エンティティ横断のネットワーク分析（循環取引）**、**財務諸表アサーションへの紐付け** ── を土台に組み込む。

## 1. レイヤ構成

| # | レイヤ | 役割 |
|---|---|---|
| L1 | データ取込・統合（ETL） | 受注・出荷・検収・請求・売上仕訳・売掛金・入金・返品/赤伝・各マスタ・契約を統合し、`SalesTransaction` に正規化。日付を分離保持（カットオフ分析の要）|
| L2 | ルールベース評価 | `config/rules/rule_catalog.yaml` の65ルールを全件に決定論的適用。各所見にアサーションを付す |
| L3 | 探索的分析 | 得意先・製品・担当・時系列・粗利の可視化で異常を洗い出す |
| L4 | 機械学習（異常検知） | Isolation Forest / LOF / Autoencoder / PCA で外れ値をスコア化。SHAP で寄与要因を提示 |
| L5 | エンティティ・ネットワーク分析 | 取引・資金フローをグラフ化し、循環経路・買戻し・関連当事者の環流を検出（売上特有）|
| L6 | 自律AIエージェント | L2〜L5 を統合し、シナリオから仮説を立て、外部証憑を自律収集して検証（`docs/agent-design.md`）|
| L7 | HITL コンソール | 人間（監査人・監査役等）が所見をレビューし、確定・棄却・是正判断を行う |

## 2. 入力・証憑・出力

### 入力（分析対象・構造化）
受注、出荷・納品、請求、売上仕訳（手動/自動の別を含む）、売掛金・エイジング、入金、返品・赤伝・値引、得意先マスタ、価格マスタ、契約、営業担当、製品マスタ。総勘定元帳との突合（完全性の裏付け）を取込時に行う。

### 証憑（実在性・妥当性の検証用、read-only）
出荷・配送確認（運送業者）、検収・納品証憑、顧客残高確認、契約条件・付随合意、入金（銀行）記録、法人登記・所在確認、反社/制裁スクリーニング、関連当事者・エンティティ関係、価格マスタ。

### 出力
統合リスクレポート、経営者・監査役向けサマリ（PDF/PPTX）、内部監査・会計監査向け証跡付き明細（Excel/CSV）、改ざん不能の監査ログ。すべての所見に根拠（違反ルール／SHAP寄与／収集証憑）とアサーションを付す。

## 3. データフロー（5フェーズ × ファネル）

```
全件 ──[L2 ルール + L4 ML + L5 ネットワーク（決定論的・低コスト）]──▶ リスクスコア + アサーション
        │
        ▼ 高リスク部分集合（例 risk_score ≥ 70 / critical 発火 / 実在性・期間帰属の high）
   [L6 エージェント: 観察→仮説生成→探索→検証→統合]  ← 外部証憑を read-only 収集
        │
        ▼
   根拠付き所見（HITL 提示）──▶ L7 人間が確定・棄却・是正判断
```

### コスト設計（重要）
全件（数百万明細）に LLM／エージェントを直接適用してはならない。順序が逆だと破綻する。L2・L4・L5 は決定論的・低コストで全件を処理し、高リスク部分集合のみをエージェントが深掘りする。この funnel が「全件網羅」と「現実的コスト」を両立させる根拠。閾値は `rule_catalog.yaml` の `thresholds`。

## 4. データ品質・完全性（全件主張の裏付け）

「全件を評価した」と言うには、母集団が完全である必要がある。取込時に (a) 売上計上額と総勘定元帳の突合（`reconciled_to_gl`）、(b) 連番・欠番チェック、(c) 期間網羅の確認、(d) 必須項目欠損の記録を行い、欠損は所見に `data_quality` フラグとして残す。完全性（completeness）は売上では網羅性の観点で特に重要（計上漏れ・簿外売上の裏返し）。

## 5. エンティティ・ネットワーク分析（売上特有）

循環取引・買戻し・関連当事者の環流は、単一明細を見ても分からない。取引相手（得意先・仕入先）と資金フローをグラフ化し、(a) サイクル検出（A→B→C→A）、(b) 得意先と仕入先の同一性、(c) 介在業者の実体・粗利・反転、(d) 資金の閉路と僅少ネット移動 を評価する。エンティティは `SalesTransaction.entity_id` と外部の関連当事者・登記情報で解決する。

## 6. マルチテナント・データ分離（プロダクト化時）

クライアント単位でデータ・モデル・監査ログを完全分離する。テナント間でデータ・学習が混ざらないこと。詳細は `docs/security-privacy.md` §5。

## 7. 拡張性

同じコア（ファネル・エージェント・HITL・監査ログ・ネットワーク分析）は他ドメインに展開できる。売上に隣接する高リスク領域 ── 仕入・購買（架空仕入・キックバック）、在庫（評価・実在性）、工事契約（進行度）、関連当事者取引全般 ── へは、`config/rules/` にドメイン別パックを追加し、`docs/agent-design.md` のツール定義に該当コネクタを足す。コアのオーケストレータ・HITL・監査ログ・アサーション枠組みは共通。
